# 数据分析工具 - 分享与部署说明

以下方式从易到难，可按对方技术水平选择。

---

## 适合后续更新推送的分发形式（推荐）

若希望**后续你改代码后能方便推送给对方**，优先用下面两种形式之一。

### 1. Git 仓库（源码更新可推送）

**做法**：把项目放到 **GitHub / Gitee / 自建 Git**，把仓库地址发给对方。

| 对方用法 | 首次 | 后续更新 |
|----------|------|----------|
| 会 Git | `git clone <仓库地址>`，然后按方式二或四装依赖、运行 | 在项目目录执行 **`git pull`**，即得到你的最新代码 |
| 不习惯 Git | 在仓库页面点「Download ZIP」下载，按方式二/四使用 | 你发新 ZIP 或他们偶尔再点 Download ZIP 覆盖解压 |

**你这边**：日常在本地改代码，`git commit` + `git push`。需要发「可下载包」时，在 GitHub/Gitee 打 **Release（标签）**，上传 zip，把 Release 链接发给对方即可。

- 优点：一次给链接，之后更新靠 `git pull` 或你发新 Release 链接；版本清晰（tag）。
- 适合：方式二（源码运行）、方式四（launcher.vbs）的用户。

### 2. 云端部署一个公网地址（你更新即所有人更新）

**做法**：用**方式三**把应用部署到 ModelScope / Render / Railway 等，只发**一个固定链接**给所有人。

- **你更新**：在 Git 里 push 新代码，云平台自动重新部署（若已连 Git），或你手动重新部署一次。
- **对方**：始终打开同一个链接，无需任何操作，打开就是最新版。

- 优点：真正的「推送更新」——你部署即他们看到更新；对方零安装、零更新操作。
- 适合：多人、外部分享，且不介意数据走公网或使用隐私模式时。

### 小结：更新如何推送

| 分发形式 | 你更新时 | 对方获得更新的方式 |
|----------|----------|---------------------|
| **Git 仓库** | `git push`（可选打 Release） | 对方在项目目录 `git pull`，或重新下载 Release zip |
| **云端部署** | 在平台重新部署/自动部署 | 对方刷新页面即可，无需操作 |
| **Docker 镜像仓库** | 构建新镜像并 push 到 Docker Hub 等 | 对方 `docker pull` 新 tag 后重新 `docker run` |
| **仅发 zip/网盘** | 重新打包、上传、发链接 | 对方需重新下载并覆盖，无自动推送 |

## 方式一：Docker（推荐，对方有 Docker 时）

**适合**：对方电脑已安装 [Docker Desktop](https://www.docker.com/products/docker-desktop/)。

**你这边**：把整个项目文件夹打成 zip 发过去，或提供 **Git 仓库地址**（便于后续更新）。

**对方操作**：
```bash
# 若用 Git：git clone <仓库地址> 后进入项目目录
# 若用 zip：解压后进入项目目录
docker build -t data-analysis-tool .
docker run -p 7860:7860 data-analysis-tool
```
然后在浏览器打开：**http://localhost:7860**。

**后续更新（若用 Git）**：在项目目录执行 `git pull`，再重新 `docker build` 与 `docker run` 即可。

- 优点：环境一致，不依赖本机 Python 版本。
- 缺点：需要安装 Docker。

---

## 方式二：源码 + Python 运行（对方会一点命令行）

**适合**：对方已安装 Python 3.10+。

**你这边**：发整个项目 zip，或**推荐**发 **Git 仓库地址**（后续更新只需对方 `git pull`）。

**对方操作**：
```bash
# 若用 Git：git clone <仓库地址> 后进入项目目录
# 若用 zip：解压后进入项目目录
pip install -r requirements.txt
python app.py
```
浏览器打开：**http://localhost:7860**。

**后续更新（若用 Git）**：在项目目录执行 `git pull`。若你改了依赖（requirements.txt），对方再执行一次 `pip install -r requirements.txt` 即可。

- 优点：无需 Docker，改代码方便；配合 Git 便于推送更新。
- 缺点：需要配好 Python 和依赖。

## 方式三：云端部署，发链接（适合多人/外部分享）

**适合**：希望对方**只打开链接**就能用，不装任何软件。

把应用部署到公网，把链接发给大家即可。

| 平台 | 说明 |
|------|------|
| **ModelScope 创空间** | 已适配端口 7860，用项目里的 Dockerfile 创建 Docker 创空间即可。 |
| **Render** | 连 GitHub 仓库，选 Web Service，Build 命令 `pip install -r requirements.txt`，Start 命令 `gunicorn -b 0.0.0.0:$PORT app:app`，并设置环境变量 `PORT`。 |
| **Railway / PythonAnywhere** | 类似思路：上传代码，配置启动命令为 `gunicorn -b 0.0.0.0:7860 app:app`（端口按平台要求改）。 |

部署成功后，把生成的 URL（如 `https://xxx.onrender.com`）发给对方。

- 优点：对方零安装，任何设备有浏览器就能用。
- 缺点：需要你注册并配置一次云服务；数据经公网，敏感数据需注意隐私或自建内网。

### 方式三可选：要求「注册后使用」

若希望**只有注册用户才能使用**（例如内部分享、控制访问），在部署时设置环境变量：

- **`ENABLE_AUTH=1`**（或 `true` / `yes`）

部署平台一般都在「服务 → 环境变量」里添加。例如 Render：Settings → Environment → Add Variable → `ENABLE_AUTH` = `1`。

开启后行为：

- 打开链接会先进入**登录页**，未登录用户无法访问分析界面和接口。
- 新用户点击「注册」填写用户名、密码（至少 6 位），注册成功后登录即可使用。
- 用户数据存在部署环境下的 SQLite 文件中（`users.db`），仅用于登录校验。

**注意**：不设置 `ENABLE_AUTH` 时与之前行为一致，任何人拿到链接即可使用，无需登录。

## 方式四：Windows 一键启动（仅本机免命令行）

**适合**：对方是 Windows、不想敲命令，但可以接受「双击一个图标」。

项目里已有 **launcher.vbs**（静默启动服务并打开浏览器）。  
它**自动识别自己所在目录**为项目目录，因此对方把整个文件夹放在任意路径（如 `D:\数据分析工具`）即可，无需改脚本。

**对方操作**：
1. 解压项目到任意目录，在该目录打开命令行，执行一次：  
   `pip install -r requirements.txt`
2. 双击项目里的 **launcher.vbs**，或右键「发送到 → 桌面快捷方式」后双击快捷方式。

之后每次双击即可启动并打开浏览器，无需再开命令行。

## 简要推荐

| 对方情况 | 建议方式 |
|----------|----------|
| **需要后续更新能推送** | **Git 仓库 + 方式二/四**，或 **方式三（你部署一个链接）** |
| 会 Docker | 方式一（Docker） |
| 会 Python / 命令行 | 方式二（源码运行） |
| 完全不想装软件、多人用 | 方式三（云端部署，发链接） |
| 仅自己/同事 Windows、想双击即用 | 方式四（launcher.vbs） |

数据仅在内存中保留约 1 小时，不落盘；若部署到公网，注意不要上传敏感数据或使用隐私模式/内网部署。
